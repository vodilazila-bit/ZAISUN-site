# ZAISUN-site — технічна документація

Сайт дитячого бренду одягу ZaiSun. Домен: zaisun.com.ua. Хостинг: GitHub Pages (репо `vodilazila-bit/ZAISUN-site`).

> **Для AI-агента:** прочитай цей файл повністю перед будь-якими змінами. Тут критичні правила, порушення яких ламає оплати на сайті.

## Архітектура

```
Покупець → index.html (SPA, каталог + кошик)
                │
                ▼
        Relay (Google Apps Script)
                │
    ┌───────────┼───────────────┐
    ▼           ▼               ▼
Google Sheets  Email-сповіщення   monobank еквайринг
(журнал)       (miroslava_zanchuk@ukr.net)   (інвойси)
```

**Файли репо:**
- `index.html` — увесь сайт (SPA): каталог, картки товарів, кошик, форма замовлення
- `data.js` — база товарів (~609 товарів, 40 категорій). Поточна версія: v13. При зміні товарів версія інкрементується
- `admin.html` — адмін-панель: редагування товарів, до 5 фото з drag-to-reorder, дублювання товару, вкладка категорій, журнал замовлень (📬), захист від конфліктів редагування (freshDataSha)
- `feed.xml` — фід для Google Merchant Center (~601 товар, оновлюється щоночі)

**Хто чим користується:** Мирослава — admin.html, Instagram, реклама. Володя — код, інтеграції, деплої.

## Relay (Google Apps Script)

- Живе під акаунтом **mirazanchuk@gmail.com** → Google Apps Script
- Версія 3. Ключові функції:
  - `monoFetch_` — обгортка для запитів до monobank з кастомним User-Agent і 5 повторами (обхід antifraud, який блокує IP Google з помилкою 403 «Change your IP address»)
  - `checkPendingPayments` — перевірка неоплачених інвойсів кожні 10 хв
  - `smartHealthCheck` — самодіагностика
- Приймає замовлення, пише в Google Sheets, шле email, створює mono-інвойси
- Журнал замовлень доступний через `admin.html` → параметр `?orders` + ключ доступу

### ⚠️ КРИТИЧНО: правила редеплою Relay

1. **НІКОЛИ не тиснути «New deployment»** — це створює новий URL і кладе всі оплати на сайті.
2. Правильно: **Manage deployments → ✏️ edit → Version: New version → Deploy**. URL залишається той самий.
3. «Who has access» завжди: **Anyone**.
4. Relay URL зашитий у **трьох місцях** — якщо URL таки змінився, оновити всюди:
   - `index.html`
   - `admin.html`
   - константа `RELAY_URL` всередині самого Apps Script

## Оплати (monobank еквайринг)

- Токен mono зберігається в Apps Script. **Не публікувати токен ніде у відкритому вигляді** (код, коміти, чати).
- Antifraud monobank блокує прямі запити з серверів Google → усі запити тільки через `monoFetch_`
- Спосіб оплати «7 тисяч до року»: MCC 5641 підтверджений
- Типова помилка 403 «Change your IP address» = antifraud; перевірити, що запит іде через monoFetch_

## Типові поломки і що робити

| Симптом | Ймовірна причина | Рішення |
|---|---|---|
| Оплата не створюється, 403 від mono | antifraud (IP Google) | перевірити, що виклик іде через monoFetch_ |
| Сайт шле замовлення «в нікуди» | Relay передеплоєний через «New deployment» → новий URL | зробити редеплой правильно АБО оновити URL у 3 місцях |
| JSON-помилка у формі замовлення | зламана структура payload після правок форми | звірити поля форми з тим, що очікує Relay |
| Конфлікт при збереженні в admin.html | хтось інший змінив data.js (freshDataSha) | перезавантажити адмінку, повторити правку |
| Товари не оновились у Google Shopping | feed.xml оновлюється щоночі | почекати до ранку або перевірити генерацію фіду |

## Правила для будь-якого агента/розробника

1. Спочатку прочитати цей README повністю.
2. Не чіпати Relay без розуміння правил редеплою (розділ вище).
3. Не виводити токени в код, коміти чи чат.
4. Перед зміною `data.js` — врахувати версіонування (інкремент версії).
5. Якщо щось працює «дивно, але працює» (наприклад monoFetch_) — це не баг, а обхід зовнішніх обмежень. Не «чистити».
